# Harness 技能 — 已知问题

## 概述

当前 SKILL.md 设计与完整实现之间的结构性差距。

---

## 问题 1 — 无评分机制：仅通过/失败

**严重程度:** 高

**问题描述:**
`harness_check.py` 仅返回 `{"passed": true/false, "errors": [...]}`。
没有量化指标衡量约束接近收敛的程度，无法比较候选方案或决定何时停止。

**修复方案:**
扩展输出以包含：

```json
{
  "heuristic_score": 0.85,
  "illegal_pass_rate": 0.0,
  "false_positives": [...],
  "false_negatives": [...],
  "passed": true
}
```

---

## 问题 2 — 无收敛判据

**严重程度:** 高

**问题描述:**
更新循环没有终止条件。`/harness-update` 可被无限调用，没有"足够好"的定义，意味着约束永远无法正式收敛，可能产生振荡。

**修复方案:**
在 SKILL.md 中添加显式收敛检查：

```
同时满足以下两个条件时视为收敛:
  - illegal_pass_rate == 0.0
  - heuristic_score   >= 0.9

硬性上限: 最多 10 次更新迭代。
若到第 10 次仍未收敛，采用观察到的最高 heuristic_score 的候选方案。
```

---

## 问题 3 — 更新触发依赖人工

**严重程度:** 中

**问题描述:**
`/harness-update` 的主要触发方式是用户手动输入命令。

**修复方案:**
定义按优先级排序的自动触发条件：

```
1. 任何 harness_check 运行后 illegal_pass_rate > 0 → 立即触发
2. 用户说"不对"/"不是我想要的"/"我们这里用X"
3. 连续两次或以上类型错误/测试失败/lint 错误
4. 用户显式输入 /harness-update
```

条件 1-3 应由 Claude Code 自主调用更新，无需等待用户显式输入。

---

## 问题 4 — 未区分误报和漏报

**严重程度:** 中

**问题描述:**
当前更新流程不区分：
- **误报（False Positives）** — 非法代码变更通过了约束检查
- **漏报（False Negatives）** — 合法代码变更被错误阻止

**修复方案:**
在编写更新提示前，对每个约束失败进行分类：

| 失败类型 | 根本原因 | harness.md 中的目标章节 |
|---|---|---|
| 误报 | 缺少禁止规则 | 添加到"禁止模式" |
| 漏报 | 过度约束 | 放松规则或添加例外 |

---

## 问题 5 — 无最大迭代次数保护

**严重程度:** 低

**问题描述:**
没有更新迭代次数上限。在极端情况下（如冲突的 lint 规则、不稳定的测试），
循环可能无限运行，或产生越来越大的 `harness.md` 导致不可读。

**修复方案:**
添加硬性上限和回退策略：

```
max_iterations: 10

达到上限仍未收敛时:
  - 采用观察到的最高 heuristic_score 的约束版本。
  - 在 harness.md 末尾追加警告块，列出未解决的问题。
  - 向用户报告："约束未完全收敛。处理所列问题后请再次执行 /harness-update。"
```

---

## 汇总表

| # | 问题 | 严重程度 | 影响组件 |
|---|---|---|---|
| 1 | 仅通过/失败 — 无评分 | 高 | `harness_check.py` 输出 |
| 2 | 无收敛判据 | 高 | SKILL.md 更新循环 |
| 3 | 更新触发依赖人工 | 中 | `/harness-update` 触发条件 |
| 4 | 误报/漏报未分类 | 中 | 精炼提示模板 |
| 5 | 无最大迭代次数保护 | 低 | SKILL.md 更新循环 |

优先解决问题 1 和 2 影响最大，因为所有其他改进都依赖于可靠的评分和明确的终止条件。
