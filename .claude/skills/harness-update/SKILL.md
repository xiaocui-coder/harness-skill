# `/harness-update` — 约束改进

基于反馈和失败结果改进约束。

约束强制层（PostToolUse 钩子运行的 `harness_check.py`）输出的验证错误
是本技能的主要自动触发信号。当钩子通过 `additionalContext` 返回
「类型错误仍然存在」等信息时，这就是「规则或验证脚本的弱点」信号。

## 步骤

### Step 1: 失败分析

从当前对话、错误消息、差异中确定「什么失败了」：

```
- 错误类型（类型错误 / 测试失败 / lint / 逻辑错误 / 风格违规）
- 失败的文件和行号
- 根本原因（规则缺失 / 规则模糊 / 脚本未覆盖）
- 来源（用户口头反馈 / PostToolUse 钩子的验证结果 / CI 失败）
```

### Step 2: 约束弱点识别

按精炼循环的思路进行思考：

> 验证返回「通过」但实际是非法操作 → 验证函数有遗漏
> 验证返回「失败」但实际是合法操作 → 验证函数过度约束

在 Claude Code 的上下文中：
- 规则中已有但生成了违规代码 → 规则说明不够明确
- 规则中未涉及的失败场景 → 需要新增规则
- 脚本未能检测到 → 需要增强脚本
- 钩子检测到了但未被修正 → 规则（描述）不足，或 additionalContext 文案不够明确

### Step 3: 约束更新

更新 `.claude/rules/harness.md` 和 `.claude/rules/harness_check.py`：
- 模糊的规则用具体示例重写
- 补充缺失的规则
- 在脚本中添加新的检查项

更新时需以注释形式留下 **为什么需要这个变更**。

脚本更新后，需 **确认与强制层的兼容性**：
- 仅添加了验证逻辑 → PostToolUse 钩子调用同一脚本，无需更改配置
- 改变了验证的 **启动命令本身**（执行路径或选项变更等）→
  需同步更新 `.claude/settings.json` 中 PostToolUse 钩子的命令
- 确认脚本的钩子模式输出（`hookSpecificOutput.additionalContext` 等）
  未被破坏（stdout 是否混入了 JSON 以外的内容）

### Step 4: 完成报告

- 变更内容（以 diff 格式简要展示）
- 新增规则的意图
- 强制层（钩子）是否有相应变更
- 提示「通过反复执行 `/harness-update` 可以持续改进约束」
